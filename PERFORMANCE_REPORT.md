# Frontend Performance Report

## 計測結果

`npm run build` の出力を比較しました。

| 対象 | 変更前 | 変更後 |
| --- | ---: | ---: |
| エントリー JavaScript | 805.85KB | 435.18KB |
| エントリー JavaScript（gzip） | 196.07KB | 120.69KB |
| CSS | 109.08KB | 110.04KB |
| CSS（gzip） | 20.09KB | 20.23KB |

## 主な要因

- `App.jsx` に複数画面の表示・状態・操作がまとまっている。
- 137件の旅先データ、画像メタデータ、関連JSONがAppから静的に読み込まれていた。
- 開発者ページ専用の品質チェック・抽選バランス診断も初期バンドルに含まれていた。
- 外部の地図・アイコンなどの大型ライブラリは使用しておらず、依存関係の重複も確認されなかった。

## 実施内容

- 旅先データを初回描画後に読み込む動的importへ変更した。
- データ準備中は抽選操作を抑え、空の候補一覧や不完全な抽選結果へ進まないようにした。
- 開発者ページを開いた時だけ、品質チェックと抽選バランス診断を読み込むようにした。
- 開発者ページのhero監査は、通常画面と同じ旅先データ上のheroメタデータを参照するよう整理した。

## 維持した仕様

- 旅先検索は正式旅先137件を対象とし、表示時は既知の豊岡市重複を1件に整理する。
- hero画像はconfirmed判定・既存の表示条件・非トリミング表示を維持する。
- 旅先データ、hero画像、画像CSS、API実装は変更していない。

## 今後の候補

- 結果詳細、比較、お気に入り、履歴、開発者ページ本体を画面単位で分離できれば、初期エントリーをさらに小さくできる。
- `App.jsx` の画面単位の分割はprops・状態の整理を伴うため、別途回帰確認を行う前提で進める。
