# AI Plan V2 Report

監査日: 2026-07-13

## 現状の問題と対応

- 従来のAIプランは長い文章をそのまま表示しており、日ごとの順序や注意点を追いにくい状態でした。
- V2ではOpenAI Responses APIへJSON Schemaを渡し、日別の予定、映えスポット、食、費用目安、雨天時の代替案、注意点を構造化して受け取ります。
- 2026-07-13に、OpenAIのJSON文字列をAPIがそのまま`plan`へ返し、クライアント側の厳しい解析に失敗した際に旧テキスト表示へ落としていた不具合を修正しました。APIは検証済みオブジェクトだけを成功応答の`plan`へ返します。
- クライアントは後方互換として、オブジェクト、JSON文字列、コードフェンス、前後説明文、二重JSON化、`plan`/`content`/`text`/`output`/`result`内のJSONを最大3段階まで安全に正規化します。
- JSONらしい値の解析に失敗した場合は生JSONを表示せず、「旅行プランの整理に失敗しました。もう一度生成してください。」と再試行導線を表示します。既存の正常プランは保持します。
- 旧形式の明確なプレーンテキストだけは「旧形式のプラン」として段落表示します。空の応答はエラーとして安全に再試行できます。
- 2026-07-14に、出力上限が日帰りでも最大1,200 tokensだったためJSONが途中で切れる問題を修正しました。日帰りは2,200、1泊2日は3,000、2泊以上は3,800 tokensを目安にし、環境変数の既定値は2,400、許容上限は4,000 tokensです。環境変数が低くても日程別の安全な必要量を下回りません。
- Responses APIの`incomplete`/`incomplete_details`とChat Completions互換の`finish_reason: length`を検知し、途中終了は`OUTPUT_TRUNCATED`として再試行へ切り替えます。切れたJSONを画面やキャッシュへ渡しません。
- 旧テキストフォールバックが概要と本文を二重描画していたため、フォールバック本文は1回だけ表示するように整理しました。成功済みキャッシュはV3へ更新し、旧形式・不正形式のキャッシュは読み捨てます。

## AIへ渡すデータ

対象旅先1件だけを送ります。`destinationId`、都道府県、市区町村、region、紹介文、タグ、季節情報、最寄り駅・駅からの補足、観光候補、ご当地グルメ、周辺候補、日程、出発地、同行者・目的、移動範囲、予算目安を含めます。

さらに、AIプラン生成を押した時だけ`photoSpots`を遅延読み込みし、confirmed映えスポットを最大3件、その名称・カテゴリ・説明・おすすめ時間/季節・天候/アクセスの注意・地図検索語とともに渡します。通常の「映え」詳細の遅延読み込み構成は変えていません。

## 出力と表示

- JSONは`title`、`concept`、`summary`、`days`、`recommendedPhotoSpots`、`localFoodSuggestions`、`budgetEstimate`、`rainPlan`、`tips`、`disclaimer`で構成します。
- APIの成功応答は`{ ok: true, plan: { ... }, model, format }`に統一します。`plan`へJSON文字列は入れません。
- 結果画面はタイトル、コンセプト、概要、日別タイムライン、映えスポット、食、費用目安、雨天案、注意点の順で、空の項目を出さずに表示します。
- 各タイムライン項目には、時刻、種別、説明、滞在目安、写真のヒント、注意、Google Maps検索導線を必要な場合だけ表示します。
- 生成中は段階的な文言を循環表示しますが、実際の進捗を示すものではありません。二重送信は画面内で防止します。

## エラーと安全性

- API未設定、混雑、タイムアウト、応答不正、その他の障害を短い日本語に置き換え、キー、スタックトレース、OpenAIの生エラー本文は一般画面に表示しません。
- 正規化は`eval`や`Function`を使わず、parse回数・長さを制限し、`__proto__`、`prototype`、`constructor`を含む値を受け入れません。
- `/api/generate-plan`はPOSTだけを受け、プロンプトを8,000文字、旅先ペイロードを12,000文字に制限します。出力上限は日程に応じて2,200〜3,800 tokens、環境変数の既定値は2,400 tokens、タイムアウトは最大30秒です。
- 本番では`OPENAI_API_KEY`をサーバー側だけで使用します。`VITE_OPENAI_API_KEY`による直呼びはlocalhost開発時だけのフォールバックです。
- 同一条件の成功済みプランは画面内キャッシュから再表示し、不要な再生成を避けます。

## レート制限の現状

画面内の二重送信防止、日次のローカル利用回数表示、入力制限、出力上限、タイムアウトはあります。一方、Vercel Serverless Functionで信頼できるIP・ユーザー単位の永続的レート制限、認証、サーバー側利用回数管理は未実装です。一般公開前には、認証設計と共有ストアを使うレート制限、予算アラートを追加してください。

## 本番確認項目

1. `OPENAI_API_KEY`と`OPENAI_PLAN_MODEL`を本番環境へ設定し、キーがブラウザのNetworkレスポンスやビルド成果物へ出ないことを確認する。
2. 実際に生成し、confirmed映えスポットが2件以上ある旅先で少なくとも2件が旅程またはおすすめ欄へ反映されることを確認する。
3. API未設定、429、タイムアウト時に安全な日本語と再試行ボタンが出ることを確認する。
4. 375px、390px、430pxでタイムライン、予算、映えスポットのカードに横スクロールがないことを確認する。
